🔄Data Flow — Server Startup
1
Render boots the Node server
Server enters startServer() async function. Nothing is available to users yet.
2
MongoDB Atlas connects
If connection succeeds, mongoRestoreToSQLite() runs — all 14 tables are pulled from MongoDB and inserted into the local SQLite DB (INSERT OR REPLACE). If MongoDB fails, server starts with empty SQLite (graceful degradation).
3
Express starts listening on port 3001
Only after MongoDB restore completes. This eliminates the race condition where requests hit empty SQLite before data is loaded.
4
Background syncs start
Two intervals: MongoDB → SQLite restore (every 3 min). Penalty cron (every 15 min): checks for 24hr warnings and 48hr/ETA penalty triggers. Report cron (every 1 hr): fires weekly report on Sundays at 8am IST if auto-send is enabled. Shopify access token also fetched and cached.
5
Real-time: after every write request
After any SQLite write (stage change, meta update, settlement, etc.), syncSQLiteToMongo() is called via setImmediate() to immediately propagate changes to MongoDB.

Tag pattern "500 partial" auto-sets advance_paid = 500, moves order to PARTIAL stage, fires advance-collection email to customer.
C

03 — Tag Mapping System
How Shopify tags drive automatic stage transitions
⚙️How applyTagMappings() Works
1
Payment type is set first
Always derived from Shopify financial_status → stored in order_meta.payment_type. Independent of tags.
2
Scan for advance pattern
Regex /(\d+(?:\.\d+)?)\s+partial/i — if any tag matches (e.g., "1000 partial"), sets advance_paid = 1000.
3
Match tag rules by priority
All tag_mappings loaded, sorted by priority ASC. First tag in order that has a matching rule wins. Stage is updated if different from current. Lower priority number = higher precedence.
4
Stage email fired on change
If stage actually changed, fireStageEmails() is called → appropriate customer + vendor emails sent immediately.
⚠️
Important: Tag Priority Must Be Managed Carefully
If two tags match the same order (e.g., "confirmed" and "transit"), the one with lower priority number wins. Incorrectly set priorities can silently demote orders (e.g., transit → confirmed). Always audit priority numbers when adding new mappings.

05 — Penalty System
How vendors are held accountable for fulfilment delays
⏱️Penalty Timeline (per order-vendor pair)
0h
Order moves to CONFIRMED or PARTIAL
stage_started_at timestamp is recorded. The penalty clock starts ticking. Penalty cron checks every 15 minutes.
24h
Warning Email Sent
Vendor receives fulfilment warning. Delay remark link sent — vendor can log a reason + commit to a new ETA date. warning_sent = 1 prevents duplicate warnings. This is the vendor's chance to communicate before penalty hits.
48h
Penalty Triggered: 48hr Breach
If vendor hasn't dispatched (stage still CONFIRMED/PARTIAL), penalty is created in order_penalties with trigger_reason = '48hr_breach'. Vendor gets penalty email. Status = PENDING (admin must confirm with amount).
ETA+1
Penalty Triggered: ETA Breach
If vendor committed to ETA in delay remarks but still hasn't shipped by that date, trigger_reason = 'eta_breach' penalty fires. Separate from 48hr penalty — a vendor can have both.
ADMIN
Admin Resolves in Penalties View
Admin reviews pending penalties. Can CONFIRM (set penalty_amount, added to next settlement) or CANCEL (waive the penalty). Vendor email sent either way.
SETTLE
Penalty Deducted in Settlement
Confirmed penalties in the settlement period are summed and added to vendor's net payable. Appears as "penalty_deduction" line in the invoice.
🔴
Risk: Penalty fires even if vendor dispatched manually (outside system)
If a vendor dispatches via WhatsApp/call and doesn't update the stage in the portal, the cron will still fire penalties at 48hr. The system only knows about stage changes in the panel. Fix: Enforce that AWB entry in the vendor portal is the only valid dispatch confirmation. Communicate this clearly to all vendors.
⚠️
No penalty for orders in NEW stage
The penalty clock only starts when order reaches CONFIRMED or PARTIAL. Orders sitting in NEW for days have no accountability mechanism. Consider adding a separate "unconfirmed order" warning — e.g., flag orders in NEW for more than 12 hours.

06 — Settlement Engine
How vendor commissions are calculated and invoices generated
🧮Commission Formula
COD / Partial Order
base      = vendor_items_revenue
commission = base × (comm_pct / 100)
gst        = commission × 0.18
invoice    = commission + gst
advance    = advance_paid (from order_meta)
net_due    = invoice − advance
type       = 'receivable' (vendor pays CrosCrow)
Prepaid Order
base       = vendor_items_revenue × 0.90
             (10% prepaid discount to customer)
commission = base × (comm_pct / 100)
gst        = commission × 0.18
net        = −(base − commission − gst)
type       = 'payout' (CrosCrow pays vendor)
ℹ️
Commission Rate Priority
1. vendor_profiles.commission_pct (per-vendor override) → 2. vendor_config.commission_pct → 3. Default 20%. Multi-vendor orders split shipping equally: each vendor's shipping charge = total_shipping / vendor_count (COD only).
📄Settlement Generation Flow
1
Admin selects vendor + date range
Only DELIVERED orders in that period are included. For multi-vendor orders: only the per-vendor stage is checked — a vendor is settled only when their items specifically are marked DELIVERED.
2
Per-order commission calculated
Each delivered order's line items belonging to this vendor are summed. Commission formula applied. Shipping split applied for COD orders.
3
Confirmed penalties added
All confirmed (not cancelled) penalties for this vendor in the same period are fetched and summed. Added to net_payable. Linked via settlement_penalties table.
4
Invoice generated
Invoice number: CC-{VENDOR}-{YYYYMM}-{timestamp}. Saved to settlements + settlement_orders tables. Status = PENDING.
5
Admin reviews + marks paid
Invoice modal shows full line-by-line breakdown with company profiles, tax calculation, penalty deduction. Admin can edit (extra discount, shipping adjustment, custom notes). Print for records.
🔴
Settlement is manual — no payment gateway integration
Net payable is calculated but actual payment (NEFT/IMPS to vendor) happens outside the system. There's no payment reconciliation or auto-payout. This means a vendor could be settled in the system but physically not paid if someone forgets. Recommendation: Add a "payment reference" field and payment date when marking invoices as paid. Future: Razorpay Payout API integration.

07 — Vendor Shopify Sync
How vendor stores connect to CrosCrow
🔗Connection Flow
1
Vendor provides credentials in Vendor Sync panel
Shop domain (e.g., mystore.myshopify.com), API Client ID, API Client Secret from their Shopify Partners app.
2
Client Credentials OAuth
Server calls POST {vendor_shop}/admin/oauth/access_token with grant_type=client_credentials. Returns scoped access token. Verified by fetching /shop.json.
3
Products browsed & imported
Admin searches vendor's Shopify catalog (title search via ?q=). Selects product → imports to CrosCrow store via Shopify Product API. Product created as DRAFT with vendor attribution.
4
Variant mapping saved
Each vendor variant ID ↔ CrosCrow variant ID mapping stored in vendor_product_mappings. This allows inventory sync: when CrosCrow variant sells, vendor's stock can be decremented.
5
Price & Stock sync
Admin can trigger sync anytime from the map modal — fetches latest price + inventory from vendor store, updates CrosCrow listings.

🔴
CRITICAL: Settlements are NOT backed up to MongoDB
The settlements, settlement_orders, settlement_penalties tables are only in SQLite. Every time Render restarts (free tier), these are wiped. Generated invoices are permanently lost. This must be fixed before generating real settlement invoices. Action: Add these three tables to SYNC_TABLES immediately.
⚠️
Audit log also not synced to MongoDB
The audit_log table is SQLite-only. All admin actions (who changed what, when) disappear on restart. For compliance and debugging, add audit_log to SYNC_TABLES or write directly to MongoDB via an AL helper.

10 — Overlapping Cases & Risk Areas
My analysis of where things can go wrong or conflict
🔴
Risk 1: Duplicate Emails on Stage Collision
If a Shopify webhook arrives AND an admin manually changes the stage at the same moment, fireStageEmails() could fire twice for the same stage transition. The customer could receive two "Order Confirmed" emails. Fix: Add a per-order, per-stage deduplication check in fireStageEmails() — track which stages have already had emails sent in order_meta.
🔴
Risk 2: Tag Sync Can Trigger Emails on Old Orders
When admin runs "Bulk Tag Sync" it re-applies all tag mappings to all 797 orders. If any order's stage changes as a result, fireStageEmails() fires. This can send confirmation/transit emails to customers of orders that were fulfilled months ago. Fix: Bulk sync should have a dry-run mode first, or skip orders in terminal stages (delivered/rto/cancelled).
⚠️
Risk 3: Penalty Clock Doesn't Reset on Hold
If an order is put on HOLD (e.g., customer requests delay), the stage_started_at is not cleared and the penalty clock keeps running. A vendor could receive a penalty for a hold that was admin-initiated, not vendor negligence. Fix: When stage changes TO hold, clear stage_started_at. When it comes back OUT of hold, reset the clock.
⚠️
Risk 4: Multi-Vendor Order — Global Stage vs Vendor Stage Confusion
Orders have both a global order_meta.stage AND per-vendor stages in order_vendor_stage. Emails fire based on global stage change. But settlement uses per-vendor stage. If admin changes global stage to "delivered" without updating individual vendor stages, customers get delivered email but vendors don't appear in settlement. Fix: When global stage changes to "delivered", auto-set all vendor stages to "delivered" unless overridden.
⚠️
Risk 5: advance_paid Tag Pattern Conflicts with Product Names
The regex /(\d+(?:\.\d+)?)\s+partial/i matches any number followed by "partial". A product tag like "10 partial pieces" or "50 partial payment" would accidentally set advance_paid = 10 or 50. Fix: Make the pattern more specific, e.g., require it to be an exact tag: tag equals "advance:500" or "partial-500" rather than free-text regex.
⚠️
Risk 6: No Idempotency on Settlement Generation
Admin can generate the same vendor + period settlement multiple times. The system doesn't prevent duplicates. You could end up with two invoices for the same period if someone clicks "Generate" twice. Fix: Add a unique constraint on vendor_name + period_start + period_end in settlements table. Or check for existing settlement before generating.
💡
Observation: JARVIS AI uses real Shopify data but no meta enrichment
JARVIS AI (Claude) fetches Shopify orders for its context but doesn't pull CrosCrow meta (stage, advance, AWB). So if you ask "how many orders are confirmed?" it uses Shopify's fulfillment_status, not our internal stage. This leads to incorrect answers. Fix: Merge CrosCrow meta into orders before passing to AI context window.

11 — Scaling & Future Roadmap
How to grow the panel to handle 10x volume
🔥 Fix Immediately (Before Scale)
1
Add settlements + audit_log to MongoDB SYNC_TABLES
Settlement data loss on Render restart is a business-critical bug. Without this, every invoice generated can disappear.
2
Add email deduplication guard in fireStageEmails()
Track sent stage-emails in order_meta (e.g., emails_sent: "confirmed,transit"). Never send the same stage email twice to the same order.
3
Upgrade Render to paid plan ($7/mo)
Free tier sleeps after 15 min. Penalty cron, sync jobs, and webhooks all fail during sleep. At current order volume (~800 orders), missed penalty windows mean real money lost.
📅 Short Term (Next 30 Days)
📱
WhatsApp / SMS Notifications
Email open rates for order updates are ~20%. WhatsApp has 80%+ read rates. Add Gupshup/Twilio as parallel channel for confirmed, transit, delivered stage messages. Template messages for customer + vendors.
🔔
Browser Push Notifications for Staff
When a new order comes in or a penalty triggers, notify staff instantly in browser. Use Push API + service worker. Eliminates need to keep the panel open all day to know what's happening.
📊
RTO Prediction Score
Using order history data: city, vendor, payment type, order value — train a simple model (or use rule-based scoring) to flag orders likely to RTO. Show score in order list. Staff can proactively call customer on high-risk orders.
⚡
Auto-Stage from Courier Webhooks
Delhivery, Shiprocket, XpressBees all offer delivery webhooks. When they push "delivered" status, auto-update order to DELIVERED stage + fire email. Currently this must be done manually.
🏗️ Medium Term (1–3 Months)
💳
Auto Payout via Razorpay
When settlement is generated and approved, trigger Razorpay Payout API to vendor's bank account automatically. Eliminate manual NEFT. Full payment reconciliation in the panel.
🏪
Vendor Self-Service Portal (Enhanced)
Currently vendors can view orders + submit AWB. Add: vendor can see their own analytics (revenue trend, RTO rate, pending settlements, penalty history). Gives vendors ownership and reduces admin queries.
🤖
JARVIS AI — Enriched with Meta
Merge CrosCrow order meta (stage, advance, AWB) into JARVIS context. Enable commands like "confirm all new COD orders from Vendor X", "generate settlement for Vendor Y for last month". True natural-language ops.
📦
Inventory Management
Currently product mappings exist but inventory sync is manual. Build auto-decrement: when CrosCrow order placed → vendor inventory decremented. Low stock alerts → flag in vendor portal + email vendor to restock.
🌟 Long Term (3–12 Months)
🌐
Multi-store Architecture
As CrosCrow grows to multiple Shopify stores (regions, brands), the panel needs to support store switching. The MongoDB + Express layer already abstracts Shopify cleanly — just parameterise the SHOP constant.
📱
Mobile App for Vendors
Vendors currently use vendor.html on mobile (not ideal). A React Native app with push notifications, one-tap AWB entry, and real-time order feed would dramatically improve vendor adoption and fulfilment speed.
📈
Customer Lifetime Value Tracking
Track repeat purchase rate, time between orders, average cart value per customer. Identify top 10% customers for loyalty programs. Currently analytics shows repeat rate % but not individual customer journeys.
🏷️
Dynamic Pricing Engine
Vendor A's product selling fast → auto-suggest price increase. Slow mover → suggest discount. Based on velocity data already available in analytics. JARVIS AI could surface these recommendations proactively.
📝 Closing Thoughts from the System

The JARVIS panel is genuinely well-designed for a 0→1 D2C operation. The dual-write MongoDB/SQLite pattern is smart — you get Render's fast cold SQLite reads with MongoDB's persistence. The tag-mapping system is a powerful abstraction that lets non-technical staff control order flow through Shopify's native UI. The penalty system creates real vendor accountability without being punitive (24hr warning before 48hr penalty).

The biggest architectural gaps are: (1) settlements not synced to MongoDB — this is a real money risk, (2) no email deduplication — one race condition and a customer gets spammed, (3) the JARVIS AI operates on Shopify data not enriched with our meta — limiting its accuracy. All three are fixable in under a day of work.

At 800 orders the system is running fine. At 5,000+ orders, the fetchAllOrders() call that loads all orders on every dashboard/analytics request will become a bottleneck (currently 4 API pages × 250 = fast, but at 5k orders = 20 pages). Plan for order pagination + caching in MongoDB before reaching that scale.
